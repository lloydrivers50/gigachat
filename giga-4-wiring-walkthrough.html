The discipline this ticket installs

  

    

      
useMessages — the hook

      
Owns state: the conversation array and isTyping. Decides when things
      happen. Never writes fetch.

    

    

      
api.ts — the network

      
Owns the HTTP call. Decides how a message reaches the server. Never
      touches React. When SSE replaces fetch later, only this file moves.

    

  






  

    ⚠ The trap — don't get stuck
    
If the fetch throws, isTyping must still go
    back to false — put it in a catch (or
    finally). Skip this and a failed request leaves the
    typing dots spinning forever. A console.error is
    enough for now; a visible error message is a later ticket.

  

  

    ◷ Noted & deferred
    
The frontend and backend currently mint different ids for the same user message. Who
    should own that id? Real question — but out of scope for GIGA-4. Leave it; it lives in
    design-notes.md. The assistant id is settled: the backend owns it (step 7).
