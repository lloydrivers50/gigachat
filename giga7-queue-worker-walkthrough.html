GIGA-7 · From one process to two

The token producer leaves the web process.

GIGA-6 streamed assistant tokens to the browser over SSE — but the producer that made those tokens still ran inside the web request. GIGA-7 lifts it out into a separate worker process. This page teaches the concepts you need to write that ticket yourself: process isolation, why an in-memory EventEmitter dies at the process boundary, and the two Redis-shaped pieces that replace it.

01

The break

lead with the wound

Here is what you shipped in GIGA-6. One Node process holds everything: the POST /messages handler that produces tokens, the GET /messages/stream SSE handler that writes them to the browser, and a single module-level EventEmitter wiring the two together.

Look at streamReply in messages.routes.ts. The thing that produces tokens — the setInterval pump — runs in the same process, on the same event loop, as the server that holds your SSE connections open:

backend/src/features/messages/messages.routes.tsyour code, today
function streamReply(message: Message) {
  const tokens = message.text.split(" ");

  const interval = setInterval(() => {        // timer fires on THIS event loop
    if (tokens.length === 0) {
      addMessage(message);
      emitDone(message.id);
      clearInterval(interval);
      return;
    }
    emitToken(message.id, tokens.shift()!);  // emit() runs listeners synchronously
  }, 120);
}

And the emitter itself — the in-process pub/sub — is just a plain Node object living at module scope in messages.stream.ts:

backend/src/features/messages/messages.stream.tsyour code, today
import { EventEmitter } from "node:events";

const emitter = new EventEmitter<Events>();  // one object, in THIS process's heap

export function emitToken(id: string, chunk: string) {
  emitter.emit("token", { id, chunk });          // calls listeners right now, on the stack
}
export function emitDone(id: string) {
  emitter.emit("done", { id });
}

The SSE handler subscribes to that same object with emitter.on("token", …) and writes each event to its held-open res. This works only because the producer and the subscriber are in the same process, looking at the same emitter variable in the same memory. That last sentence is the entire fragility we're about to expose.

Why this is a problem at all

The echo finishes in ~360ms, so it feels fine. But the ticket's whole point: a real travel agent doesn't echo — it calls train APIs, waits on taxis, runs an LLM. Run that inside the web process and a slow agent run blocks the event loop for every other request, a crash mid-reply takes the web server down with it, and you can't scale "thinking" separately from "serving." The work has to leave this process.

See the loop the timer fires on

Before we split anything, get eyes on the machine. Node runs your JavaScript on a single thread driven by an event loop. The loop has phases; two matter here. The timer / macrotask queue is where setInterval callbacks wait their turn. The microtask queue (resolved Promises, queueMicrotask) is drained completely after each macrotask, before the loop moves on. Step it:

The one fact this stepper exists to burn in

emitter.emit("token", …) is synchronous. It is not queued. When the timer callback calls emitToken, the emitter immediately walks its listener list and calls each listener on the current call stack — including the SSE handler's res.write() — before emit returns. There is no scheduling, no await, no boundary crossed. That directness is exactly why it only works inside one process: it's just a function call reaching into the same heap.

The kill shot: now there are two processes

GIGA-7 moves streamReply into a separate worker process. The instant it moves, the picture below is what you actually have — two Node processes, each a full, independent OS process: its own heap, its own event loop, its own copy of every module-level variable. The worker has its own emitter; the web process has a different emitter. They are not the same object. They cannot be.

The problem statement, crisp

A Node worker is not a thread sharing the parent's memory — it is a separate OS process with its own address space. One process literally cannot see another process's in-memory objects. So the in-process EventEmitter, which worked precisely because everyone shared one heap, cannot bridge two processes. GIGA-7 needs something that lives outside any single process — reachable by both. That something is Redis.

02

Redis — the thing that lives outside every process

the shared backbone

Chapter 1 ended with a need: something both processes can reach that belongs to neither of them. That is exactly what Redis is.

Redis is an in-memory data store that runs as its own server process, listening on a TCP port (6379 by default). Any process that opens a socket to it can read and write the same data. It is not a library you embed; it is a separate program you talk to over the network — even when "the network" is just localhost.

The mental swap

The EventEmitter was a shortcut that only worked because publisher and subscriber shared a heap. Redis replaces it with a real, out-of-process broker: publisher and subscriber now share a server, not a heap. Two different processes, both holding a socket to the same Redis, are looking at the same data — which is precisely what the in-memory object could never offer once they split.

Redis will wear two hats in GIGA-7

Don't deep-dive either yet — just plant the flag. One shared Redis backbone serves both jobs:

(a) durable backing store for the job queue

When the web process accepts a message, it writes a job into Redis. The job sits there, durably, until the worker picks it up — even across restarts. BullMQ (Chapter 3) is the queue library built on this.

(b) pub/sub transport for fanning tokens back

When the worker produces a token, it publishes it to a Redis channel; the web process's SSE handler is subscribed and writes it onward. Redis Pub/Sub (Chapter 5) replaces the EventEmitter's fan-out.

Stand it up with one compose service

You don't install Redis by hand — you declare it. A single service in a new docker-compose.yml at the repo root gives both processes something to connect to:

docker-compose.yml  (new)illustrative
services:
  redis:
    image: redis:7          # pinned major — same store both processes dial
    ports:
      - "6379:6379"         # expose so localhost web + worker can connect
Why it belongs in compose, not in code

Redis is infrastructure, not application logic — it has a lifecycle independent of your Node processes (it should already be up before either starts, and stay up across their restarts). Declaring it in compose means docker compose up -d is the one command that gives both your processes a backbone to dial, with nothing baked into the app's source.

03

BullMQ — the queue (web → worker)

Redis hat (a): the job

Redis gives you a shared store. BullMQ is a Node job-queue library built on Redis — it turns "write a job, read a job later, from another process" into a clean API so you don't hand-roll list operations and locking.

It has two sides. This chapter is the producer side (the web process). The consumer side is Chapter 4.

conceptwhat it iswhere in gigachat
Queuenew Queue(name, { connection }) — a named list of jobs, persisted in Redisnew queue/messages.queue.ts
queue.add()writes one job { id, text } to Redis and returns immediatelycalled from POST handler
connectionan ioredis connection (BullMQ wants maxRetriesPerRequest: null)new queue/connection.ts

Following the same instinct as emitToken — keep the produce API in one place — you'd wrap the .add() in a thin enqueueReply:

backend/src/queue/messages.queue.ts  (new)illustrative — not the full file
import { Queue } from "bullmq";
import { connection } from "./connection";

const replyQueue = new Queue("reply", { connection });

// one place every .add() lives — same instinct as emitToken
export function enqueueReply(id: string, text: string) {
  return replyQueue.add("reply", { id, text });   // job persisted in Redis, returns now
}

Now the change to messages.routes.ts: the POST handler stops doing the work. It still validates, still persists the user message immediately, still mints the assistant id and returns 202 { id } on the wire — but instead of calling streamReply(), it enqueues a job and returns. streamReply leaves this file entirely.

backend/src/features/messages/messages.routes.tsthe shape of the change
messagesRouter.post("/messages", (req, res) => {
  const { text } = req.body;
  // …validate text… (unchanged)
  addMessage(userMessage);            // persist USER msg first (failure-mode case 1)

  const id = crypto.randomUUID();
  res.status(202).json({ id });        // SAME wire response as GIGA-6

  streamReply(assistantMessage);      // ← deleted: the producer moves out of this process
  enqueueReply(id, text);             // ← enqueue a job, then return. No slow work here.
});
The problem BullMQ solves

It decouples "accept the request fast" from "do the slow work." A bare in-process function call (today's streamReply) or a setInterval ties the work's lifetime to the web request and the web process. A queue job lives in Redis — it survives the web process returning, restarting, even crashing; another process can pick it up at its own pace. That durability across processes and restarts is the thing a function call fundamentally can't give you.

Run note

The queue is useless without somewhere to live — it needs the redis service from Chapter 2 already running. docker compose up -d first, then the web process.

04

The worker — a separate Node process

the consumer

A job is sitting in the queue. Something has to pick it up and do the work. That something is a BullMQ Worker, and it runs in its own Node process — a brand new entry point, started by its own command.

A new Worker(name, processor, { connection }) connects to the same Redis, watches the named queue, and runs your async processor function for each job it pulls. The streamReply logic — the body that was in the routes file — moves here. This is its new home.

backend/src/worker.ts  (new — its OWN process entry point)illustrative — not the full file
import { Worker } from "bullmq";
import { connection } from "./queue/connection";
import { emitToken, emitDone } from "./features/messages/messages.stream";
import { addMessage } from "./features/messages/messages.store";

new Worker("reply", async (job) => {
  const { id, text } = job.data;          // the {id, text} POST enqueued
  const reply = `Echo: "${text}"`;
  const tokens = reply.split(" ");

  // the old streamReply body, relocated — same emit calls, new process
  for (const word of tokens) {
    emitToken(id, word);
    await sleep(120);
  }
  addMessage({ id, role: "assistant", text: reply });
  emitDone(id);
}, { connection });
Reinforce — what a worker process actually is

This is not a thread, not a callback, not a "background task" inside the web server. It is a full Node process with its own memory, its own heap, and its own event loop — started by its own command (e.g. npm run worker, or its own container). It happens to import the same source files, but at runtime it is a separate copy of every module-level value. The web process and the worker share nothing in memory — only Redis.

Walk straight into Chapter 1's wall

Notice the worker calls emitToken(id, word) — and emitToken still does emitter.emit(...) on the module-level emitter. But this is the worker's emitter, in the worker's heap. The SSE handler is subscribed to the web process's emitter. They are different objects (exactly the two-heap picture from Chapter 1). So right now these tokens go nowhere — they emit into a process with no SSE listener. The res you need to write to lives in the other process. Chapter 5 fixes precisely this.

05

Pub/sub the tokens back

the seam payoff

The worker now produces tokens. But the SSE res — the held-open connection to the browser — lives in the web process. The worker cannot res.write(); that res isn't in its memory. So how do tokens get from worker back to the right SSE connection?

Redis Pub/Sub. The worker publishes each token to a Redis channel; the web process's SSE handler is subscribed to that channel and writes whatever it receives to res. Redis sits in the middle exactly where the EventEmitter used to — except it's reachable from both processes.

Here's the move that makes the whole ticket pay off: you reimplement emitToken / emitDone to publish to Redis instead of the local emitter — keeping the same signatures and the same payload shape:

backend/src/features/messages/messages.stream.tssame API, new transport
import { EventEmitter } from "node:events";
import { connection } from "../../queue/connection";

const CHANNEL = "tokens";
const pub = connection.duplicate();   // publisher connection

// SAME signature as GIGA-6 — callers (the worker) don't change
export function emitToken(id: string, chunk: string) {
  pub.publish(CHANNEL, JSON.stringify({ type: "token", id, chunk }));
}
export function emitDone(id: string) {
  pub.publish(CHANNEL, JSON.stringify({ type: "done", id }));
}

And the SSE handler in messages.routes.ts swaps emitter.on() for a Redis subscribe — but everything downstream of that, the res.write(\`event: …\\ndata: …\\n\\n\`) framing, is byte-for-byte identical to GIGA-6:

backend/src/features/messages/messages.routes.ts  · GET /messages/streamillustrative
messagesRouter.get("/messages/stream", (req, res) => {
  // …same text/event-stream headers + flushHeaders() as GIGA-6…

  const sub = connection.duplicate();      // DEDICATED subscriber connection (see gotcha)
  sub.subscribe(CHANNEL);
  sub.on("message", (_chan, payload) => {
    const msg = JSON.parse(payload);
    // SAME SSE wire format as GIGA-6 — the frontend can't tell anything changed
    res.write(`event: ${msg.type}\ndata: ${JSON.stringify(msg)}\n\n`);
  });

  req.on("close", () => {            // same cleanup discipline as GIGA-6, one layer down
    sub.unsubscribe(CHANNEL);
    sub.quit();
  });
});
The seam payoff — say it out loud

Everything behind the SSE boundary changed: in-process function → cross-process queue + worker + pub/sub. But emitToken/emitDone kept their signatures, and the bytes on the SSE wire are identical. So useMessages.ts — the new EventSource("/messages/stream"), the addEventListener("token", …) that does msg.text + chunk, the "done" listener that clears streaming — does not change a single line. That's the seam from GIGA-6 finally earning its keep.

Two mastery subtleties — get these exactly right

1. Redis pub/sub is fire-and-forget. A published message is delivered only to subscribers connected at that moment. There is no persistence, no buffering, no replay. If an SSE client is mid-reconnect when a token publishes, it misses that token permanently. This is fine on purpose — it's exactly GIGA-6's deliberately-deferred "reconnect-resume / Last-Event-ID" scope. (Note the contrast with the queue: a dropped job is unacceptable and BullMQ persists it; a dropped token is acceptable and pub/sub doesn't. Different guarantees, on purpose.)

2. A subscribed connection is dedicated. Once a Redis client issues SUBSCRIBE, that connection enters subscriber mode and can't run normal commands (it can only sub/unsub). So the subscriber must be a separate connection from the publisher and from the BullMQ queue — hence connection.duplicate() for each SSE client's subscriber, torn down on req.on("close").

one message's whole journey

Every box has been motivated and placed. Here is the complete picture — the same boxes you watched get assembled, now all wired — followed by the trace of a single message coming full circle back to the user.

Close the loop

Everything behind the SSE boundary was swapped — an in-process function call became a cross-process queue + worker + pub/sub spanning two heaps and a separate Redis server. And the browser never knew. useMessages.ts sees the same 202 {id}, the same event: token frames, the same done. The seam held. That is why GIGA-6 built the seam before GIGA-7 needed it.

07

Running it

compose + two processes

One process became three moving parts: Redis, the web server, and the worker. Today's single npm run dev isn't enough — you now start the worker as its own process too, both pointed at the same Redis.

docker-compose.yml  (consolidated)illustrative
services:
  redis:
    image: redis:7
    ports: ["6379:6379"]   # both web & worker dial this

  # web and worker can stay as plain `npm run` processes for local dev,
  # or become their own services here later — each is a SEPARATE process
backend/package.json  · scriptsthe shape of the change
"scripts": {
  "dev":    "tsx watch src/server.ts",   // the WEB process (unchanged entry)
  "worker": "tsx watch src/worker.ts"    // the WORKER process (new entry)
}

Three terminals, in order:

local dev — three terminalsillustrative
# 1 — infra: bring up the shared backbone first
docker compose up -d           # Redis listening on :6379

# 2 — the web process (serves HTTP, holds SSE open)
cd backend && npm run dev

# 3 — the worker process (consumes jobs, produces tokens)
cd backend && npm run worker
The "aha" to watch for in the logs

Send a message and watch which terminal does what. The web terminal returns 202 before the worker has logged a single token — acknowledge fast on one process, answer slow on another. Seeing the producer/consumer split in two separate terminals is the whole point of the ticket made visible.

One thing to observe (don't fix yet)

Kill the worker mid-reply and restart it: BullMQ retries the job (at-least-once delivery — the job was durable in Redis). But the pub/sub tokens already sent are gone (fire-and-forget). Watch what the frontend sees. That gap is the seam for later failure-mode work — note it, don't solve it here.

GIGA-7 teaches toward, but does not implement, the queue+worker rewrite. Snippets are illustrative — you write the ticket. The seam from GIGA-6 is what makes the swap invisible to the frontend.
